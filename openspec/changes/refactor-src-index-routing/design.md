## Context

当前 `general-server/src/index.ts` 作为应用路由入口，只负责引入模块路由并统一挂到 `/api` 下，但内部仍然存在 `router.use(siteMenuRouter)` 这种裸挂载方式。与此同时，最新的后端约束已经明确要求：模块路由接入 `src/index.ts` 时，必须使用业务名前缀统一挂载，例如 `router.use('/site-menu', siteMenuRouter)`。

当前 `siteMenu` 模块内部已经同时存在两个层次的路径：

- 兼容查询入口 `/getMenu`
- 模块内部 CRUD 路径 `/site-menu`

因此这次重构不仅是改一行入口代码，还需要明确入口层与模块层的职责边界，避免出现双重前缀、职责重叠或兼容接口被无意破坏的问题。

## Goals / Non-Goals

**Goals:**

- 让 `src/index.ts` 中的模块路由接入方式全部改为“业务名前缀挂载”
- 明确入口层只负责业务前缀，模块层只负责模块内部相对路径
- 收敛 `siteMenu` 模块路由，避免入口层和模块层重复承担路径设计职责
- 保持已有接口在可控范围内兼容，避免不必要的对外破坏
- 为后续新增业务模块建立统一接入模式

**Non-Goals:**

- 不在本次变更中新增新的业务模块
- 不在本次变更中重构 controller、service、repository 业务逻辑
- 不对数据库结构、DTO 结构或返回契约做额外修改
- 不强行把所有历史兼容路径一次性删除

## Decisions

### 决策 1：`src/index.ts` 统一负责业务名前缀挂载

入口文件 `src/index.ts` 只负责两件事：

- 引入模块路由
- 以业务名前缀注册模块路由

例如：

- `router.use('/site-menu', siteMenuRouter)`

这样做优于裸挂载的原因：

- 路由边界更清晰，入口层一眼可见所有业务域
- 后续新增模块时有稳定模式，不会再出现路径散落
- 更符合当前 skill 的硬性约束

### 决策 2：模块 `router` 改为声明相对路径

当模块被入口层挂载到业务前缀后，模块内部 router 应优先声明相对路径，而不是继续重复业务名前缀。

以 `siteMenu` 为例：

- 入口层负责 `/site-menu`
- 模块层内部负责 `/`、`/:id`、`/getMenu`

这样做优于继续在模块内部写 `/site-menu/...` 的原因：

- 避免入口层和模块层重复拼路径
- 模块可迁移性更好，挂载点变化时无需大改模块内部代码
- 更符合 router 的职责定义

### 决策 3：保留必要的兼容查询入口

对于已有的语义化兼容接口，例如 `GET /api/getMenu`，允许保留；但其是否保留，应由入口层显式声明，而不是依赖模块裸挂载碰巧暴露出来。

这意味着重构后需要明确：

- 业务主挂载点走 `/site-menu`
- 历史兼容入口如 `/getMenu` 需显式保留或转发

选择显式保留而不是隐式继承的原因：

- 兼容性来源更清楚
- 后续清理历史接口时更容易定位
- 不会因为模块内部调整导致兼容路径意外消失

## Risks / Trade-offs

- [风险] 重构入口层后，现有接口路径可能出现重复前缀或路径变化
  → Mitigation：在设计中固定“入口层负责业务前缀、模块层负责相对路径”的边界，并补接口测试

- [风险] 历史兼容路径 `/getMenu` 如果处理不当，可能被重构误删
  → Mitigation：将兼容路径作为显式设计对象，而不是依赖当前偶然行为

- [风险] 后续其他模块若继续照旧裸挂载，会重新破坏入口一致性
  → Mitigation：本次变更同时把接入模式沉淀为明确规范和实现样例

## Migration Plan

1. 调整 `src/index.ts`，将 `siteMenu` 路由改为业务名前缀挂载
2. 调整 `siteMenu.router.ts`，使其内部以相对路径组织模块接口
3. 显式处理 `GET /api/getMenu` 兼容入口
4. 运行集成测试，确认重构前后的接口可用性

回滚策略：

- 若入口重构导致兼容接口异常，可回滚 `src/index.ts` 与 `siteMenu.router.ts` 的路由组织改动
- 由于本次不涉及数据库和业务逻辑改造，回滚范围仅限路由层

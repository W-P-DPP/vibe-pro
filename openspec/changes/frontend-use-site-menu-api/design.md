## Context

当前前端目录页由 `frontend-template/src/data/tool-directory.ts` 提供静态 `toolSections`，`AppLayout` 和 `HomePage` 直接消费该数据。后端已经提供 `GET /api/site-menu/getMenu`，返回树形菜单节点，字段主要包括 `id`、`name`、`path`、`icon`、`sort`、`children`、`remark`。  
前端现有展示模型除了 `name/path/icon` 之外，还依赖 `summary`、`status`、`note` 等展示字段，因此在接入接口时需要一层明确的前端数据转换，而不是直接把后端 DTO 暴露给页面组件。

## Goals / Non-Goals

**Goals:**

- 让前端目录数据统一来源于后端 `siteMenu` 接口
- 保证侧边栏目录与首页卡片区使用同一份菜单状态，避免重复请求和渲染不一致
- 为菜单接口接入补齐加载态、失败态、空态，确保页面在接口异常时可控退化
- 收敛前端请求配置，使开发环境可直接访问 `/api/site-menu/getMenu`

**Non-Goals:**

- 本次不改造后端 `siteMenu` 接口结构
- 本次不新增真实业务子页面路由，只处理目录数据来源切换
- 本次不扩展三级以上目录的复杂视觉层级，仍以“顶级分组 + 分组内项目”方式呈现

## Decisions

### 决策 1：由布局层统一获取菜单数据并向页面共享

在 `AppLayout` 附近新增菜单状态容器（例如 provider、context 或专用 hook 承载的共享状态），由布局层发起一次 `GET /api/site-menu/getMenu` 请求，再把结果同时提供给侧边栏和首页内容区。

这样做优于“各组件分别请求”的原因：

- 避免侧边栏和首页重复发请求
- 避免一处成功、一处失败导致界面状态不一致
- 更符合当前 `AppLayout + Outlet` 的页面组织方式

备选方案是让 `AppLayout` 与 `HomePage` 独立请求，但会引入重复加载和状态分裂，因此不采用。

### 决策 2：增加前端菜单适配层，将后端树形结构转换为现有目录展示模型

后端返回的是 `SiteMenuResponseDto[]`，前端页面需要的是“分组 + 项目卡片”结构，因此新增显式适配层：

- 顶级菜单节点映射为目录分组
- 子菜单节点映射为分组内项目
- `remark` 优先映射为分组说明或项目摘要
- 当项目缺少摘要时，使用统一中文兜底文案
- 当 `path` 为 `http/https` 地址时视为可直达入口，否则视为占位入口

这样做优于直接在组件中拼装字段的原因：

- 转换规则集中，组件只负责展示
- 后续如果后端字段扩展，调整点更集中
- 可以继续兼容现有 `HomePage` 与侧边栏的展示结构

### 决策 3：前端请求层默认改为使用本地 `/api`，开发环境通过 Vite 代理到后端

当前 `src/api/request.ts` 默认请求 `jsonplaceholder.typicode.com`，与实际仓库联调目标不一致。本次改为前端默认请求 `/api`，开发环境在 `vite.config.ts` 中增加代理到本地后端服务地址，仍保留 `VITE_API_BASE_URL` 作为覆盖配置。

这样做优于继续保留远程默认地址的原因：

- 本地开发无需额外改代码即可直接联调后端
- 请求模块职责更清晰，默认值与当前仓库业务一致
- 兼容部署环境通过环境变量覆盖网关地址

## Risks / Trade-offs

- [风险] 后端接口缺少 `summary/status` 等展示字段
  → Mitigation：在前端适配层补齐派生规则和中文兜底文案，避免页面直接依赖缺失字段

- [风险] 当前前端展示结构天然偏向两层目录，而后端实体支持更深层级
  → Mitigation：本次先约束为“顶级分组 + 子项列表”，更深层级暂不作为本次范围

- [风险] 接口异常会让首页核心区域不可用
  → Mitigation：补充加载态、失败态和空态，并保证失败时侧边栏与首页都能给出一致反馈

- [风险] 变更请求基地址可能影响现有演示接口
  → Mitigation：保留 `VITE_API_BASE_URL` 覆盖能力，并同步梳理示例请求模块

## Migration Plan

1. 新增前端 `site-menu` API 模块、类型定义和适配层
2. 将 `AppLayout` 和 `HomePage` 从静态 `toolSections` 切换为共享菜单状态
3. 调整请求基础地址和开发代理配置，确保可访问本地后端
4. 通过构建和必要的前端验证确认目录渲染与异常状态可用

回滚策略：

- 若接口接入后出现异常，可先回滚前端菜单接入改动，恢复静态 `toolSections`
- 请求基地址与代理配置可独立回退，不影响后端菜单接口本身

## Open Questions

- 是否需要把后端 `remark` 作为前端项目说明的唯一来源，还是允许前端继续保留少量本地补充映射
- 若后端后续出现三级目录，前端是折叠展示、平铺到卡片区，还是单独新增二级分组视觉

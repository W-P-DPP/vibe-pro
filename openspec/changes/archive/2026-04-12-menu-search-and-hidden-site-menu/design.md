## Context

当前 `frontend-template` 会在 `AppLayout` 中一次性拉取 `siteMenu` 树，并把结果传给侧边栏和 `HomePage` 渲染；右上角搜索图标只是静态装饰，没有搜索状态、输入框或结果面板。`HomePage` 的卡片打开逻辑已经区分普通链接与 `strict` 菜单，但没有任何“隐藏菜单”概念。

后端 `general-server/src/siteMenu/*` 已具备完整六层结构，菜单数据来自 `sys_site_menu` 和 `siteMenu.json` 导入链路。当前实体、DTO、仓储查询、服务校验和响应契约都没有 `hide` 字段，因此前端无法稳定获得隐藏状态，也无法通过新增/更新接口维护该字段。

本次变更同时触达前端头部交互、前端菜单筛选策略、前端环境变量、后端菜单数据模型、CRUD 契约和菜单导入链路，属于跨模块联动变更，适合先用设计文档固定关键决策。

## Goals / Non-Goals

**Goals:**

- 在 `AppLayout` 头部搜索图标点击后展开搜索输入框，并在输入过程中展示前端菜单模糊匹配结果
- 保持搜索完全在前端完成，不新增后端搜索接口
- 在前端通过 `hide` 字段默认隐藏菜单，并通过环境变量控制的彩蛋口令解锁隐藏菜单搜索结果
- 在后端 `siteMenu` 模块中新增 `hide` 布尔字段，并向查询、创建、更新、导入链路完整透传
- 保持现有 `strict` 打开逻辑、站内/站外打开逻辑、菜单路由命名和整体响应结构不变

**Non-Goals:**

- 不把“隐藏菜单”设计成权限控制或安全机制
- 不新增独立的菜单管理前端页面
- 不改造后端为搜索提供模糊查询接口
- 不在本次变更中新增基于 `hide` 的角色、鉴权或服务端过滤逻辑
- 不调整现有 `HomePage` 搜索引擎卡片区域的功能

## Decisions

### 决策 1：头部菜单搜索保持纯前端实现，复用已加载的菜单树

搜索能力直接建立在 `AppLayout` 已加载完成的菜单树之上，不新增接口，也不在用户输入时触发重新请求。搜索索引由前端根据当前菜单树实时构建，匹配字段至少包含菜单名称、备注、路径以及所属分组名称。

这样做的原因：

- 用户已经明确要求“查询在前端实现即可”
- 当前菜单数据量较小，前端内存过滤足够，避免引入额外接口和请求抖动
- 与现有 `AppLayout` 的一次性加载模式一致，改动面可控

备选方案：

- 新增后端模糊查询接口：实现更重，也与需求不符

### 决策 2：搜索结果只聚合可操作菜单项，分组节点仅作为上下文展示

搜索结果的可点击项以“最终可打开的菜单项”为主。对于 `HomePage` 中“有 children 的分组节点”，搜索时展开其叶子节点参与匹配；对于“没有 children 但自身带 path 的节点”，视作可操作项直接参与匹配。结果中额外展示其所属分组名称，帮助用户理解落点。

这样做的原因：

- 当前首页真正可打开的是卡片项，不是纯分组标题
- 隐藏菜单如果不在常规目录中渲染，搜索结果必须直接可打开，不能只滚动到不存在的分组锚点
- 结果模型更稳定，后续可直接复用现有 `openTool` 行为

备选方案：

- 搜索结果只返回分组并滚动定位：对隐藏菜单不成立，且命中后仍需二次找卡片

### 决策 3：`hide` 语义由前端消费，后端查询接口返回完整树和 `hide` 字段

后端查询接口继续返回完整菜单树，但每个节点都显式包含 `hide` 布尔字段，不在服务端过滤隐藏节点。前端正常渲染目录时过滤 `hide=true` 的菜单，搜索框命中彩蛋口令后才把隐藏菜单纳入结果列表。

这样做的原因：

- 用户要求搜索在前端实现，隐藏菜单也必须在前端可被解锁，因此前端必须拿到原始隐藏状态
- 只返回可见菜单会导致彩蛋无法显示隐藏菜单
- 服务端保持“数据提供者”角色，避免把彩蛋逻辑写入后端

权衡：

- 隐藏菜单仍会下发到前端，因此 `hide` 只能表示“默认不展示”，不是权限边界

备选方案：

- 后端默认过滤隐藏菜单，再增加单独参数或接口解锁：会把彩蛋逻辑引入接口层，与需求不符

### 决策 4：彩蛋口令使用 `VITE_SITE_MENU_HIDDEN_KEYWORD`，按去空格后大小写不敏感精确匹配

前端新增 `VITE_SITE_MENU_HIDDEN_KEYWORD` 环境变量，默认值为 `dpp`。搜索框会对输入做 `trim().toLowerCase()` 归一化；当归一化后的输入与配置值完全一致时，隐藏菜单结果区解锁，否则隐藏菜单继续不可见。

这样做的原因：

- 用户明确要求将词写入环境变量，方便后续修改
- 精确匹配比“包含即触发”更可控，避免普通搜索词误触彩蛋
- 大小写不敏感更符合口令输入容错预期

备选方案：

- 硬编码 `dpp`：后续改词需要改代码
- 模糊包含匹配触发彩蛋：误触风险高

### 决策 5：后端 `hide` 字段按 `strict` 的既有扩展方式贯穿实体、DTO、仓储、服务、导入与测试

`hide` 作为 `siteMenu` 的一等布尔字段，加入 `SiteMenuEntity`、请求/响应 DTO、仓储查询字段、服务校验和种子/导入节点定义。创建请求在缺省时默认 `false`，更新请求保持部分更新语义，只在显式传入时修改；旧种子和旧导入文件未提供 `hide` 时统一兼容为 `false`。

这样做的原因：

- 与仓库里已存在的 `strict` 字段扩展方式一致，改造路径明确
- 不支持导入/更新会造成字段契约残缺，后续维护成本更高
- 默认值为 `false` 可以平滑兼容现有 `siteMenu.json` 和已有接口调用

备选方案：

- 只改查询和创建，不改更新/导入：字段生命周期不完整，后续仍需补改

## Risks / Trade-offs

- [风险] 隐藏菜单会被完整下发到前端，具备前端调试能力的用户仍可看到相关数据
  → Mitigation：在设计和规格中明确 `hide` 只是展示控制，不承担权限或安全语义

- [风险] 搜索入口放在 `AppLayout` 头部，状态管理如果与目录加载耦合过深，容易影响现有布局稳定性
  → Mitigation：搜索状态局限在头部组件层，复用现有 `menuTree` 数据，不改目录加载时机

- [风险] 隐藏菜单默认从侧边栏和首页列表移除后，若搜索结果仍只支持滚动定位，点击会失效
  → Mitigation：搜索结果统一复用现有打开逻辑，直接打开菜单目标，而不是依赖页面锚点

- [风险] 后端仅新增数据库字段而遗漏 `siteMenu.json`/导入链路，会导致重导入后数据丢失 `hide`
  → Mitigation：把实体、DTO、仓储、服务、JSON 种子、文件导入和测试作为同一批变更

## Migration Plan

1. 在后端 `siteMenu` 实体和数据库表中新增 `hide` 布尔字段，默认值为 `false`
2. 扩展后端 `siteMenu` DTO、仓储查询、服务校验、创建/更新/查询响应和文件导入链路，保证 `hide` 全链路可读可写
3. 更新 `general-server/siteMenu.json`，让种子结构兼容 `hide`
4. 在前端新增 `VITE_SITE_MENU_HIDDEN_KEYWORD` 配置，并在 `AppLayout` 中实现搜索框、结果面板和隐藏菜单解锁逻辑
5. 让侧边栏和首页目录默认只消费可见菜单，搜索结果按需包含隐藏菜单
6. 补齐前端与后端相关测试，覆盖默认隐藏、彩蛋解锁、查询字段透传与非法类型校验

回滚策略：

- 前端回滚时移除搜索框和 `hide` 过滤逻辑，恢复静态搜索图标
- 后端回滚时停止读写 `hide` 字段并回退 DTO/响应契约；数据库字段可暂时保留为兼容列，不要求同步删表字段

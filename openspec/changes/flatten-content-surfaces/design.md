## Context

当前前端已经使用 `shadcn` 组件和 token 驱动的主题体系，但首页仍然通过外层 `max-width`、内容外边距、圆角和阴影建立卡片感。用户希望把内容区改为更贴边、更平的结构化布局，这会同时影响全局 token、主内容容器、首页分组容器以及入口卡片的表面样式。

约束如下：

- 必须继续复用现有 `shadcn` 组件体系，不能引入平行设计语言。
- 必须保留搜索、目录、分类和入口行为，不把样式调整扩展成信息架构调整。
- 必须在浅色与深色主题下都呈现一致的平面化结构。
- `frontend-template/design.md` 要求界面克制、低噪音、结构清晰，因此这次改动是进一步收敛装饰层，而不是增加新表现。

## Goals / Non-Goals

**Goals:**

- 让首页主内容区尽量不使用额外外边距，内容直接贴近主布局可用区域。
- 去除首页内容分组和入口卡片上的圆角与阴影。
- 用边框、间距和标题分组替代卡片化装饰来表达层级。
- 让浅色与深色主题都使用同一套平面容器逻辑。

**Non-Goals:**

- 不修改搜索逻辑、目录逻辑、入口数据结构或跳转行为。
- 不重构全站所有组件的圆角和阴影，只覆盖首页内容相关区域。
- 不更换当前 `shadcn` 组件库或引入新的布局框架。

## Decisions

### 1. 通过页面级容器覆写实现“平面化”，而不是全局删光所有 token

首页内容区、分组容器和入口卡片将直接在页面层使用 `rounded-none`、`shadow-none` 和更贴边的布局类，而不是把整个主题的 `--radius` 与 `--shadow-*` 全局清零。

原因：

- 用户要求聚焦于“内容区”，不是整个项目所有控件。
- `Button`、`Input`、`Badge` 等 `shadcn` 基础控件仍然可能保留少量圆角，更符合渐进式调整。
- 页面级覆写风险更低，便于后续继续收敛或局部回退。

备选方案：

- 全局重置 radius/shadow token。未采用，因为影响范围过大，会把无关组件也一起扁平化。

### 2. 主内容区取消宽度居中外边距，改为依附 `SidebarInset` 可用宽度

首页根容器将不再使用显著的左右外边距与居中卡片布局，而是改成更贴近主内容区边界的分组流式布局。

原因：

- 直接响应“内容去不设置外边距”的要求。
- 能更清楚地表达“工作台内容区”，而不是“居中的展示卡片”。

备选方案：

- 保留 `max-width` 只减少 `px`。未采用，因为仍会保留明显的内容外边距感。

### 3. 用分隔线和标题行维持层级，不依赖阴影

分组容器和入口项将改为依赖边框、分割线、标题行和稳定间距来表达层级；hover 态最多只改变边框和背景，不再通过阴影抬升。

原因：

- 这更符合 design.md 的低噪音、结构优先原则。
- 深色主题中阴影表达弱且容易显脏，去掉后更一致。

备选方案：

- 用更深的背景块代替卡片。部分采用，但不会增加新的装饰色块层级。

## Risks / Trade-offs

- [页面显得过于硬朗] → 保留清晰的边框、分组标题和足够的留白，避免内容拥挤。
- [局部组件圆角仍存在，导致风格不完全统一] → 将无圆角要求限定在“内容区容器与卡片”，避免全站大范围破坏。
- [深色主题层级不足] → 使用边框对比、标题行和背景层次而不是阴影来补足区分度。

## Migration Plan

- 先在首页与布局层实现平面化容器样式。
- 在浅色与深色主题下验证可读性、边界感和焦点态。
- 如出现层级不足，再微调边框与背景对比，但不恢复圆角和阴影。
- 回滚方式：恢复 `HomePage.tsx`、`AppLayout.tsx` 与相关 token/类名中的圆角、阴影和内容外边距样式。

## Open Questions

- “不设置外边距”是否仅指首页主内容区，还是后续还要扩展到侧栏外的全部页面？
- 搜索栏、按钮和 badge 这类基础控件是否也要继续收敛到无圆角，还是仅内容容器需要平面化？

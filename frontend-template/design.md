# 前端设计规范

## 0. 主题来源

- 主题风格由用户定义，不由本规范预设固定视觉风格。
- 当用户提供一套 `shadcn` 主题样式、token、配色、圆角、阴影或组件外观要求时，用户提供的主题规范优先级最高。
- 本文档中关于视觉气质、色彩倾向和主题表达的描述，仅在用户尚未提供明确主题方案时作为默认约束使用。
- 后续实现应优先围绕用户提供的 `shadcn` 主题进行落地，而不是坚持当前文档里的默认审美倾向。

## 1. 设计原则

- 产品整体应采用克制的商务视觉语言。
- 界面气质应偏冷静、高效、可信，而不是装饰化、活泼化或玩具化。
- 所有视觉决策优先保证清晰度、层级感与一致性，而不是追求新奇。
- 禁止过度设计，功能简单的区域不应叠加多余的视觉概念、装饰层和表现性包装。
- 无论是展示页、仪表盘、表单页还是数据密集型页面，都必须保持统一的设计气质。

## 2. 视觉基调

- 在没有用户自定义主题方案时，默认基调为冷淡、克制、专业。
- 在没有用户自定义主题方案时，优先使用低饱和的中性色与冷色系，例如灰、石板灰、冷蓝及受控的强调色。
- 当用户提供明确的 `shadcn` 主题风格后，视觉基调、色彩倾向和氛围表达以用户方案为准。
- 无论采用哪套主题风格，都应避免无节制的装饰堆叠、过量模糊和影响信息识别的视觉噪音。

## 3. 色彩系统

- 所有颜色必须通过语义化 token 管理，包括 `background`、`surface`、`text`、`text-muted`、`border`、`primary`、`accent`、`success`、`warning`、`destructive`。
- 组件代码中不应随意引入零散的十六进制颜色，除非是在定义或调整设计 token。
- 文本、边框、焦点态、状态反馈必须保证足够对比度。
- 强调色应被克制地用于主操作、选中态、焦点提示和关键数据强调，不能泛滥使用。

## 4. 排版规范

- 继续使用 `Geist Variable` 作为主要西文字体。
- 中文回退字体保持为 `PingFang SC`、`Microsoft YaHei` 或等效系统字体。
- 排版应传达秩序感与专业度，强调稳定的层级而不是装饰性。
- 标题要简洁、紧凑、明确，避免过度表现化。
- 正文优先保证可读性与节奏一致性，不做风格化实验。

## 5. 布局规范

- 布局应强调结构、对齐和可预测的间距。
- 桌面端页面应以清晰的内容分区、适度的卡片使用和统一的宽度约束为主。
- 移动端应收敛为单列的清晰阅读流，但不能破坏层级关系。
- 间距、圆角、边框粗细与阴影必须通过统一 token 控制，不能零散定义。
- 同一信息层级只保留必要的容器层级，避免为了“设计感”重复套卡片、描边、背景块或说明模块。
- 允许较高的信息密度，但绝不能出现拥挤、混乱或难以判断主次的界面。

## 6. 组件规范

- 当前项目以现有 `shadcn` 组件作为基础 UI 系统。
- 允许在现有 `shadcn` 组件基础上进行二次开发。
- 二次开发必须保持当前 UI 风格，不得衍生出另一套平行的组件语言。
- 自定义变体必须兼容项目既有的 token 体系、间距体系、圆角体系和交互体系。
- 新的业务组件应尽量由现有基础组件组合而成，而不是重复造轮子。

## 7. 交互规范

- 交互应快速、明确、低噪音。
- 动效的作用是帮助理解状态变化，而不是制造装饰感。
- 可以使用轻量过渡处理状态切换、悬停反馈、内容显现和面板移动。
- 避免滥用微交互、弹性动画或戏剧化效果。
- 对于可交互元素，必须在适用时明确提供悬停、焦点、按下、禁用、加载、空状态和错误状态表现。

## 8. 双主题策略

- 浅色主题和深色主题都必须作为一等公民设计。
- 两套主题必须表达同一套品牌气质、层级关系、间距逻辑和组件语言；若用户提供主题方案，则以该方案定义的一致性为准。
- 主题切换后不能出现“像两个产品”的视觉割裂。
- 若用户提供了明确的浅色/深色主题样式，应按该方案实现，不得自行追加另一套主观风格解释。
- 任一新组件或页面，只有在双主题下都验证通过，才算设计完成。

## 9. 内容呈现

- 数据、状态和操作信息的可扫描性必须高于装饰内容。
- 标题、筛选区、表格、图表和表单分组应保持稳定且一致的层级。
- 空状态和辅助说明应简洁、可执行、偏操作化，不应写成宣传文案。
- 业务文案表达应直接、专业、克制。
- 不要为了填满页面而添加无业务价值的说明区、概念标签、统计块或装饰性文案。

## 10. 响应式规范

- 桌面端优先服务于高效的信息访问和多区域扫描。
- 平板与移动端需保留相同的层级结构，只减少同时出现的区域数量。
- 溢出、换行、吸顶区域和操作区位置都必须被显式设计，不能依赖默认行为。
- 移动端点击区域必须足够可用，但不能因此让桌面端显得臃肿。

## 11. 可访问性

- 键盘导航必须可用于导航、弹窗、菜单、表单和复合组件。
- 焦点态在浅色和深色主题下都必须清晰可见。
- 文本与界面对比度必须满足正常工作环境下的可读性要求。
- 信息传达不能只依赖颜色。

## 12. 禁止事项

- 禁止引入第二套无关的 UI 风格。
- 禁止在用户已经提供 `shadcn` 主题方案后，自行改写成另一套主观视觉风格。
- 禁止把偏暖、生活方式、手作感的审美混入核心商务界面。
- 禁止绕过 token 体系随意定义阴影、圆角和间距。
- 禁止用大量强调色掩盖层级设计不足的问题。
- 禁止为了显得“更完整”而堆叠无必要的卡片、图标、渐变、说明文案、占位统计和装饰模块。
- 禁止交付在浅色与深色主题下气质明显不一致的组件。
